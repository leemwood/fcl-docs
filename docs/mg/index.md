| 选项大类 | 细分选项 | 功能作用（《我的世界》Java版场景） | 适用设备/场景 | 风险/注意事项 | 相互影响 |
|----------|----------|------------------------------------|---------------|---------------|----------|
| 最大GLSL缓存大小 | 自定义数值（如4096MB） | 缓存编译后的光影着色器代码，数值越大，切换Shaderpack、加载光影的速度越快，减少重复编译卡顿 | 中高配安卓设备（内存≥6GB）、频繁切换光影的场景 | 数值过大会占用过多运行内存，导致MC因内存不足闪退；建议按内存比例设置（6GB内存设2048MB，8GB+设4096MB） | 与“光影/Shaderpack启用”强相关：缓存越大，光影加载速度提升越明显；与其他驱动/版本设置无直接关联 |
| | -1（禁用缓存） | 不缓存着色器，每次启动/切换光影都重新编译 | 低配安卓设备（内存≤4GB）、测试光影兼容性的场景 | 加载光影速度慢，首次进入世界可能卡顿30秒以上 | 仅影响光影加载效率，与其他设置无关联 |
| 启用ANGLE作为OpenGL ES驱动 | 尽可能不启用 | 优先调用手机原生OpenGL驱动，仅在原生驱动完全不兼容时才触发ANGLE转译 | 原生驱动优化好的设备（如骁龙8系、天玑9系）、纯生存无光影场景 | 高版本光影（如BSL v8、SEUS PTGI）可能直接闪退，不支持计算着色器相关特性 | 1. 决定“ANGLE Depth Clear修复方案”是否生效（仅ANGLE启用时该修复有用）；2. 影响“OpenGL版本”上限（原生驱动可能不支持4.5+）；3. 不支持依赖ANGLE的光影/mod |
| | 尽可能启用 | 优先使用ANGLE转译层，原生驱动兼容则自动回退，平衡兼容性与性能 | 多数安卓设备、兼顾光影和流畅度的通用场景 | 转译层会带来约5%-10%的性能损耗，低配设备帧率略有下降 | 1. 支持“ANGLE Depth Clear修复方案”生效；2. 提升高版本GL（4.3+）和计算着色器扩展的兼容性；3. 与“MultiDraw模拟方案”协同（ANGLE下Indirect方案更稳定） |
| | 不启用 | 强制关闭ANGLE，全程使用原生OpenGL驱动 | 原生驱动稳定的老设备、运行低配整合包的场景 | 无法运行依赖ANGLE的光影/mod，部分新版本MC（1.19+）可能出现渲染错误 | 1. “ANGLE Depth Clear修复方案”完全失效；2. 高版本GL（4.5+）和计算着色器扩展易兼容失败；3. 仅支持原生驱动兼容的MultiDraw方案（如DrawElements） |
| | 启用 | 强制开启ANGLE，将桌面OpenGL指令转为手机OpenGL ES指令 | 中高配设备、运行路径追踪光影/高级渲染mod（如Create）的场景 | 必须搭配OpenGL 4.3+版本使用，否则会出现驱动冲突；部分小众GPU可能不兼容 | 1. 强制“ANGLE Depth Clear修复方案”可生效；2. 仅支持GL 4.3+版本（低版本GL会触发冲突）；3. 优化“MultiDraw Indirect/Compute方案”的兼容性；4. 是计算着色器扩展生效的核心前提 |
| OpenGL报错设置 | 自动 | 由MobileGlues根据当前渲染环境，自动选择报错处理策略（流畅优先则忽略轻微错误，调试优先则记录错误） | 新手用户、不确定当前光影/mod兼容性的场景 | 自动模式可能掩盖轻微渲染错误（如偶尔的区块闪烁），不易排查问题 | 与“光影/mod兼容性”强相关：忽略报错时，可兼容部分有轻微瑕疵的光影；记录/严格模式下，易触发高版本GL/扩展的报错 |
| | 不忽略报错 | 严格校验每一条OpenGL指令，出现错误立即记录到日志并触发提示 | 调试光影/mod兼容性、排查闪退/花屏问题的场景 | 错误校验会增加CPU负担，导致帧率下降5%-15%；严重错误会直接终止游戏 | 1. 放大“不完整扩展”“高版本GL”的兼容性问题（易触发报错）；2. 与“MultiDraw实验性方案”冲突（实验性方案易触发错误校验） |
| | 忽略shader/program报错 | 跳过着色器编译、程序链接阶段的非致命错误，允许光影/mod继续运行 | 光影存在轻微代码瑕疵但仍可使用的场景（如民间修改版光影） | 可能出现局部画面异常（如光影水反失效、天空变黑） | 仅缓解“光影加载报错”，与驱动/版本/扩展设置无直接关联 |
| | 忽略shader/program/framebuffer报错 | 跳过着色器、程序、帧缓冲三层非致命错误校验，最大化避免渲染卡顿/闪退 | 强行运行高版本光影的低配设备、测试非常规光影的场景 | 可能掩盖严重的渲染错误（如区块黑块、模型透明失效） | 可部分抵消“高版本GL”“不完整扩展”的兼容性问题，但会掩盖“ANGLE深度缓冲异常”等硬件层面错误 |
| MultiDraw模拟方案 | 自动 | 自动检测GL版本和驱动兼容性，选择最优的MultiDraw模拟方式 | 多数用户的默认选择、无需手动调试的场景 | 自动选择的方案不一定是性能最优的，追求极限帧率需手动调整 | 1. 受“OpenGL版本”影响（GL 4.3+支持Compute/Indirect方案，低版本仅支持BaseVertex/DrawElements）；2. 受“ANGLE驱动”影响（ANGLE下Indirect方案更稳定，原生驱动下BaseVertex更兼容） |
| | 尽量使用Indirect方案模拟 | 采用间接渲染指令模拟MultiDraw接口，减少CPU向GPU发送指令的次数，提升区块渲染效率 | 中低配设备、运行大型整合包（如RLCraft）的场景 | 部分老驱动可能不支持间接渲染指令，导致画面闪烁 | 1. 需GL 4.3+版本支持（低版本开启自动回退）；2. 与ANGLE驱动协同性好，原生驱动下兼容性略差 |
| | 尽量使用BaseVertex方案模拟 | 基于顶点偏移的模拟方式，对GL 4.3以下的老版本驱动适配性最优，也可在高版本GL中使用 | 老款安卓设备（GPU为Mali-G72及以下）、运行1.16及以下版本MC的场景 | 性能提升幅度有限，高版本光影下效果不明显 | 1. 兼容所有GL版本，不受驱动类型影响；2. 与“高版本GL+光影”搭配时，性能不如Indirect方案 |
| | 尽量使用MultiDraw Indirect方案模拟 | 专用的间接多绘制指令模拟，适配高版本GL（4.5+）的高级特性 | 中高配设备、运行带高清材质+光影的场景 | 对GL版本要求高，低于4.5版本会失效并自动回退 | 1. 仅支持GL 4.5+版本（低版本开启无效）；2. 需搭配ANGLE驱动或高性能原生驱动（否则易闪退） |
| | 强制使用DrawElements方案模拟 | 采用最基础的逐元素绘制指令模拟，兼容性拉满 | 驱动极度老旧的设备、其他方案均闪退的场景 | 性能最差，大型世界中帧率会明显下降 | 1. 不受GL版本、驱动类型影响（全兼容）；2. 与“高负载场景（大型基地/高清材质）”冲突（帧率暴跌） |
| | 尽量使用Compute方案模拟（实验性） | 基于计算着色器的模拟方式，理论上CPU开销最低、渲染效率最高 | 高配设备（骁龙8 Gen2/天玑9300+）、测试实验性光影的场景 | 属于实验性功能，可能出现光影断层、粒子消失等bug；仅支持GL 4.3+版本 | 1. 需GL 4.3+版本+计算着色器扩展开启+ANGLE驱动启用（三者缺一则失效）；2. 与ANGLE驱动协同性好，原生驱动下bug较多；3. 与“路径追踪光影”搭配时性能最优 |
| 自定义目标OpenGL版本 | 不启用 | 不手动指定版本，使用设备原生支持的最高GL版本 | 新手用户、追求原生稳定性的纯生存场景 | 原生版本可能无法满足高版本光影/mod的需求，导致功能缺失 | 1. 限制“扩展启用”（原生版本低于4.3则无法开启计算着色器扩展）；2. 影响“MultiDraw模拟方案”选择（原生版本低则仅支持基础方案）；3. 与ANGLE驱动协同（ANGLE可提升原生版本的兼容性） |
| | OpenGL 4.6/4.5 | 向MC暴露最高级的桌面GL版本，解锁路径追踪光影、复杂粒子效果、高清材质等全部渲染特性 | 顶配安卓设备（GPU为Adreno 750/740/Mali-G7xx，内存≥8GB）、画质优先的场景 | 对设备GPU驱动要求极高，低配设备强行开启会闪退或帧率低于10FPS | 1. 支持所有扩展（计算着色器、direct_state_access等）开启；2. 必须搭配ANGLE驱动（原生驱动极少支持4.6）；3. 优化“MultiDraw高级方案”“FSR1”的兼容性；4. 与低配设备硬件冲突（易闪退） |
| | OpenGL 4.4/4.3 | 主流中高版本GL，支持绝大多数现代光影（如BSL、Complementary Reimagined）和渲染mod | 中高配安卓设备（内存6-12GB）、兼顾画质与流畅度的场景 | 部分极端光影（如SEUS PTGI）仍无法运行，需搭配ANGLE使用 | 1. 支持计算着色器扩展开启（4.3核心特性）；2. 不支持MultiDraw Indirect方案（该方案需GL 4.5+，低版本开启会自动回退）；3. 与ANGLE驱动搭配可兼容更多光影；4. 低版本扩展（如timer_query）无冲突 |
| | OpenGL 4.2/4.1及以下 | 暴露低版本GL，关闭高级渲染特性，优先保证游戏运行稳定性 | 低配安卓设备（内存≤4GB）、运行老旧整合包（1.12.2及以下）的场景 | 无法运行任何现代光影，部分新mod（如Iris 1.20+）会直接报错 | 1. 所有高级扩展（计算着色器、direct_state_access）无法开启；2. 仅支持MultiDraw基础方案（BaseVertex/DrawElements）；3. 与ANGLE驱动搭配无意义（无法解锁高级特性） |
| ANGLE Depth Clear修复方案 | 不启用 | 关闭深度缓冲修复逻辑，减少驱动层面的性能损耗 | ANGLE驱动下无画面异常的设备、追求极限帧率的场景 | 部分GPU（如Mali-G68）关闭后会出现区块边缘黑块、天空渲染错误 | 1. 仅当“ANGLE驱动启用”时才有意义（原生驱动下无效）；2. 与“光影启用”强相关（光影场景下黑块问题更明显） |
| | 启用方案#1 | 修复ANGLE驱动下的深度缓冲清除异常，解决区块黑块、光影断层、天空变黑等画面问题 | 开启ANGLE后出现画面异常的设备、运行光影的场景 | 仅在ANGLE启用时生效，关闭ANGLE后该选项无意义；会带来轻微性能损耗 | 1. 强制依赖“ANGLE驱动启用”（否则无效）；2. 与“高版本GL+光影”搭配时，修复效果更明显；3. 与其他修复方案无冲突（仅此一种深度缓冲修复） |
| 启用不完整的OpenGL43扩展 | 开启 | 强制解锁部分OpenGL 4.3的核心特性（如compute shader），让依赖该特性的光影/mod能够运行 | 设备原生GL版本为4.2，但需运行4.3光影/mod的场景 | 属于“强行解锁”，可能出现渲染错误、闪退等兼容性问题；仅建议调试时使用 | 1. 需“自定义GL版本≥4.2”（低于4.2开启无效）；2. 与“ARB_compute_shader扩展”强相关（开启后可解锁计算着色器基础功能，无需重复开启该单独扩展）；3. 与ANGLE驱动搭配可降低兼容风险；4. 与“不忽略报错”冲突（易触发大量报错） |
| | 关闭 | 不启用不完整的扩展，避免驱动层面的兼容性冲突 | 追求稳定运行的场景、无需4.3特性的纯生存玩法 | 无法运行依赖4.3扩展的光影/mod | 1. 不影响其他扩展（如timer_query、direct_state_access）；2. 与低版本GL搭配更稳定 |
| 启用不完整的ARB_compute_shader扩展 | 开启 | 启用计算着色器扩展，支持路径追踪光影、复杂粒子效果的计算渲染逻辑 | 运行SEUS PTGI等路径追踪光影的场景 | 需搭配GL 4.3+版本和ANGLE驱动使用，否则无效；低配设备开启后帧率暴跌 | 1. 强制依赖“GL版本≥4.3”+“ANGLE驱动启用”（缺一不可）；2. 与“MultiDraw Compute方案”协同（性能翻倍）；3. 与“路径追踪光影”强相关（无此扩展无法运行）；4. 与低配设备硬件冲突（帧率暴跌） |
| | 关闭 | 禁用计算着色器扩展，减少驱动负担 | 纯生存玩法、低配设备场景 | 无法运行任何依赖计算着色器的光影/mod | 1. “MultiDraw Compute方案”自动失效（回退其他方案）；2. 与其他扩展无冲突；3. 与低版本GL搭配更稳定 |
| 禁用推荐的timer_query扩展 | 开启 | 关闭timer_query扩展，解决部分设备因该扩展导致的帧率波动、卡顿、闪退问题 | 开启后出现帧率不稳定的设备、流畅度优先的场景 | 会导致模拟器底层的渲染性能统计（如帧耗时）失效，但不影响游戏/光影的帧率显示功能 | 1. 与其他扩展无直接冲突；2. 与“高版本GL+光影”搭配时，帧率稳定效果更明显；3. 与“调试性能”冲突（无法查看驱动层面帧耗时） |
| | 关闭 | 启用timer_query扩展，恢复原生驱动的性能统计功能 | 需调试驱动层面性能的场景、设备无帧率波动的场景 | 部分老GPU（如Adreno 610）会出现帧率骤降的问题 | 1. 与老GPU驱动冲突（易帧率暴跌）；2. 与“不忽略报错”搭配时，可能触发扩展相关报错 |
| 启用高级的direct_state_access扩展 | 开启 | 启用直接状态访问扩展，优化GPU渲染状态切换效率，减少CPU开销，提升高负载场景（如大型基地）的帧率 | 运行大型整合包、建造大型基地的场景 | 仅支持GL 4.5+版本，低版本开启无效果；部分驱动可能不兼容 | 1. 强制依赖“GL版本≥4.5”（低于4.5开启无效）；2. 与“高负载场景（大型基地/高清材质）”协同（帧率提升明显）；3. 与ANGLE驱动搭配兼容性更好；4. 与其他扩展无冲突 |
| | 关闭 | 禁用直接状态访问扩展，避免高版本扩展带来的兼容性问题 | 低配设备、驱动老旧的场景 | 高负载场景下CPU开销增加，帧率略有下降 | 1. 不影响其他扩展；2. 与低版本GL搭配更稳定 |
| （实验性）启用内置的FSR1 | 开启 | 通过FSR1超分技术，对MC整体渲染画面做超分优化，低分辨率下提升画面清晰度 | 低配设备、低分辨率运行MC的场景 | 实验性功能，可能出现画面模糊、边缘锯齿等问题；对整体画面生效，非仅特定光影 | 1. 与其他设置无强制依赖（独立生效）；2. 与“高版本GL+光影”搭配时，可能叠加画面模糊（需调整FSR1强度）；3. 与“低分辨率设置”协同（低分辨率下效果更明显） |
| | 关闭 | 禁用FSR1超分技术，使用原生分辨率渲染 | 追求原生画质的场景、中高配设备 | 无额外风险，为默认稳定选项 | 1. 不影响其他设置；2. 与“高清材质”搭配更适合（保留原生清晰度） |
